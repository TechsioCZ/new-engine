# apps/medusa-order-dashboard-plugin Agent Guardrails

This package is the internal Medusa v2 plugin that contributes the unified order dashboard Admin UI route. It is not a standalone admin app and it is not a fork of the Medusa Admin dashboard. The Medusa app in `apps/medusa-be` only registers and loads this plugin.

## Scope Default

- Default write scope is `apps/medusa-order-dashboard-plugin`.
- Admin UI route code belongs under `src/admin/**`, currently `src/admin/routes/order-dashboard/**`.
- `apps/medusa-be` is in scope only for plugin registration, the workspace dependency, existing API endpoint compatibility checks, or explicitly approved backend endpoint/workflow work.
- Do not move this dashboard back into `apps/medusa-be/src/admin` unless the user explicitly reverses the plugin decision.
- Treat `apps/admin` as historical/reference material for this ticket. Do not implement the delivery path there.
- Treat `libs/ui` as out of scope for ordinary plugin work. Medusa Admin extensions should use Medusa Admin-compatible packages first: `@medusajs/ui`, `@medusajs/icons`, `@medusajs/admin-sdk`, `@medusajs/js-sdk`, `react-i18next`, and React Query.
- If a backend gap is found, document the exact missing endpoint, request shape, response shape, and blocked UI behavior before changing backend code.

## Required Sources

Before implementing or changing this dashboard, inspect these sources in this order:

1. Current plugin source under `apps/medusa-order-dashboard-plugin/src/admin`.

4. Existing Medusa Admin customizations under `apps/medusa-be/src/admin` only as compatibility/reference.

   - `.codex/skills/building-admin-dashboard-customizations`
   - `.codex/skills/building-with-medusa` only when backend work is explicitly in scope.

When official docs, existing backend behavior, and ticket exports disagree, preserve current project behavior unless the mismatch is a proven bug or a documented product decision changes it.

## Plugin Architecture

- The plugin must export Admin customizations through the package `./admin` export generated by `medusa plugin:build`.
- `apps/medusa-be/medusa-config.ts` should register the plugin by package name, not by relative source path.
- Keep plugin source committed. Do not commit generated `.medusa` output or `node_modules`.
- Use same-origin Admin API calls from inside Medusa Admin. Prefer `baseUrl: "/"` with session auth over standalone app environment variables such as `VITE_MEDUSA_BACKEND_URL`.
- Keep the route under `src/admin/routes/order-dashboard/page.tsx` unless a broader feature module is justified.
- If route code grows into multiple concerns, extract feature-local helpers under `src/admin/routes/order-dashboard/` or `src/admin/features/order-dashboard/`.

## UI Rules

- Build a dense, operational, scannable Medusa Admin surface. Do not add marketing layouts, decorative hero sections, or standalone admin shell navigation.
- Filters and row actions must be visibly and semantically separated.
- Table behavior must cover loading, empty, error, pagination, sorting where supported, filtering where supported, selection, and stable selected/eligible counts.
- Prefer Medusa UI primitives for Admin consistency.
- TanStack Table or VTable may be introduced only after documenting why Medusa UI primitives cannot support the required table behavior without fragile code. Add packages through package-manager commands and verify the plugin build.
- Do not use `@techsio/ui-kit` or `libs/ui` inside this plugin unless the task explicitly expands scope and the Medusa Admin bundling contract is verified.
- Every visible string belongs in the plugin Admin i18n resources. Do not leave English literals in Czech UI, and do not copy broken Notion mojibake into final labels.

## Code Organization

- Keep API fetchers, query keys, mutation helpers, types, and formatters close to the dashboard route unless they become shared by another plugin route.
- Keep business rules pure and testable where practical.
- Do not duplicate backend business logic in React. The frontend may calculate display state, but authoritative mutation rules must stay server-side.
- Page components should render state and coordinate interactions; endpoint contracts, formatting, and eligibility helpers should live in separate local modules.
- Shared order DTOs, query keys, i18n namespaces, and endpoint contracts need one owner per change. Do not let parallel work rewrite the same shared surface independently.

## Backend Gap Rule

Backend work is justified only when at least one is true:

- The current Admin API cannot express the required mutation or read model.
- Frontend scanning would be incorrect because pagination or filtering cannot preserve the business rule.
- A workflow must be atomic, audited, or permission-checked server side.
- A custom project integration already exposes backend semantics that should not be duplicated in React.

When backend work is justified, keep it as a separate, explicit scope. Backend changes should normally land under `apps/medusa-be/src/api/admin`, `apps/medusa-be/src/workflows`, or related utilities, not inside this plugin.

## Planning

Plan files for this plugin live in `apps/medusa-order-dashboard-plugin/agent-plans`. Use `plan-graph` with explicit dependency overlays from `apps/medusa-order-dashboard-plugin/agent-plans/README.md`; do not assume a pile of `.plan.md` files has an implicit execution order.

For current-lane status use `dag` against the same plan selection and dependency overlay. Keep generated graph snapshots under `apps/medusa-order-dashboard-plugin/plan-graphs` so they remain local-only unless intentionally promoted.

## Verification

For plugin code changes, run the narrowest relevant checks:

- `pnpm.cmd --dir apps/medusa-order-dashboard-plugin run typecheck`
- `pnpm.cmd --dir apps/medusa-order-dashboard-plugin run build`
- `pnpm.cmd exec biome check --write <changed plugin files>`

For plugin registration or Admin integration changes, also verify the Medusa app can consume the plugin:

- `pnpm.cmd --dir apps/medusa-be exec medusa build --admin-only`
- `pnpm.cmd --dir apps/medusa-be run test:e2e:admin:built` when browser smoke coverage is needed and dependencies are available.

This workspace expects Node `>=24` for `apps/medusa-be`. If local Node is older, record that as an environment constraint instead of hiding build failures.

## Commits and Scope

- Keep the plugin UI source, minimal `apps/medusa-be` plugin registration, and matching `pnpm-lock.yaml` changes together.
- Do not mix unrelated `apps/admin`, `libs/ui`, backend workflow, or storefront changes into the dashboard plugin commit.
- Ignore unrelated dirty files outside the active scope, especially local env files, `.refact/**`, generated `.medusa/**`, and ignored local plan/image exports.
